## 🎯 一句话介绍

这是一个**完整的 BTC 交易工作流技能**——不只是获取数据，而是从分析到建议，从警报到执行，从入场到止盈止损，全流程自动化。

说直白点，就是让你的智能体每天帮你看盘、写报告、盯关键价位，甚至帮你开仓设止盈止损——你只要躺着等飞书消息就好。

**你的智能体用它，可以：**
- 📊 每天自动写日报，发到你手机上（不用你再问"现在 BTC 多少钱了")
- 🔔 24 小时监控市场，关键价位触发即时分析（睡觉时也在盯盘）
- 💰 分析后自动开仓、设止盈止损（可选，不想实盘就纯看报告）
- 📦 每轮交易建议独立管理，结束时自动归档（不纠缠上一单的输赢）

## ✨ 核心亮点

### 🔥 全流程闭环

```
获取数据 → 技术分析 → 生成报告 → 发送飞书
    ↓
给出建议 → 监控市场 → 触发执行 → 更新状态 → 归档周期
```

这不是一堆散装脚本拼起来的"工具箱"，而是一条完整的生产线——每一步都自动衔接，你不用操心"数据拿到了怎么分析""分析完了怎么通知我"。

### 📊 数据获取的差异化设计

三种任务，三种数据，三种脚本——不共用、不冗余、各取所需：

| 任务类型 | 数据脚本 | 获取什么数据 | 设计意图 |
|---------|---------|-------------|---------|
| **日报任务** | `get_enhanced_analysis.js` | 30天日线 + 14根4h K线 + 恐惧贪婪指数 + 期权数据 + 交易侧数据（资金费率、OI、多空比、Taker比） | 完整的市场全貌，从宏观到微观，从价格到情绪，给你最全面的判断依据 |
| **即时分析** | `get_instant_data.js` | 12根4h + 4根1h + 8根15m K线 + 实时 ticker + 交易侧数据 | 精细的短周期数据，捕捉触发瞬间的市场细节，回答"为什么现在触发？" |
| **警报器** | `api.js` | 实时价格 + 分钟级 K线 + 交易量 | 足够轻量，国内直连 CryptoCompare，无代理也能跑，只检查条件不深度分析 |

为什么要分开？因为三种任务的需求完全不同：
- 日报需要**广度**——30天历史、期权市场、恐惧指数，才能判断大方向
- 即时分析需要**精度**——15分钟级别、实时 ticker，才能解释"此刻发生了什么"
- 警报器需要**速度**——轻量请求、快速判断，而不是每次检查都拉一大堆数据

**不该用日报脚本做警报检查，不该用警报脚本写即时分析。** 各司其职，才能高效。

每次模型给你的建议都不一样？今天让他做空，明天他又叫你做多？

模型给出的建议格式每次都不一样？"请告诉我这一单适合开多少倍率！"——你每次都要重复这句话？

你甚至不知道模型为什么会给出这个建议？每次都要追问"仔细跟我讲讲"？

**如果你也遇到过这些情况，这个技能就是为你准备的。**

我们设计了一套完整的提示词工程，确保：
- 每一次分析都是同样严谨的标准流程
- 每一份报告都包含完整的技术面解读
- 每一个建议都附带清晰的理由说明
- 每次任务都把该做的事情做完：获取数据、回顾历史报告、给出建议、设定警报...

你需要做的，就是说一句："七月，做一次日报分析"——剩下的他自己会搞定。

### ⚡ 24/7 警报器引擎

如果你想让智能体每分钟检查一次价格，你会怎么做？用心跳轮询？

问题是——上下文会越堆越多，token 烧得飞快，而且每次心跳都要走一遍完整的推理流程，性价比实在太低。

**PM2 常驻进程才是正确的解法。**

警报器引擎独立运行，不占用智能体的上下文，用 Node.js 定时器以分钟级间隔检查市场条件。触发时才唤醒智能体做即时分析——平时就安静地待在后台。

更妙的是，警报不是死板的固定监控。智能体可以根据当前行情**即时设计警报规则**：
- 看到斐波那契 0.618 回撤位很关键？那就设一个价格警报在那儿
- 发现多空比异常重要？那就加一个多空比监控警报
- 判断今天会有大波动？那就设一个振幅预警

**警报随行情而生，随完成而灭——这才是灵活的触发时机管理。**

### 💰 实盘交易执行（v4.12 新增）

分析完了就完了吗？能不能真正开仓？

当然可以——而且我们设计了好几层安全护栏，让你睡得安稳：

- **固定 3x 杠杆**：不给你机会加杠杆赌运气
- **逐仓模式**：亏了就亏这一单，不会连环爆仓
- **止盈止损全覆盖**：开仓就必须设 TP/SL，不留裸仓
- **整数位偏移**：止盈价自动避开整数关口（72000 → 72478），防止差一点不到

放心，这不是"分析完就给你乱开仓"的冒险玩具——每一步都有约束，每单都在可控范围内。

### 🔄 SPAWN 隔离机制 + 分阶段日报

你是否遇到过这种情况：聊得越来越多，模型开始"降智"了？

上下文堆积是 LLM 的硬伤——对话越长，注意力越分散，到最后连最简单的指令都执行不好。

**我们已经解决了这个问题。**

每次日报、每次即时分析，都在**独立的新会话**中执行：
- 主会话保持干净，不会被历史数据污染
- 分析会话从零开始，注意力永远在线
- 上一轮分析的细节不会影响下一轮判断

**v5.0 新升级：日报分四阶段执行**

更进一步，日报任务拆分为四个独立阶段：
```
阶段一：数据获取 → 返回 → 主会话 SPAWN 阶段二
阶段二：市场分析 → 返回 → 主会话 SPAWN 阶段三
阶段三：仓位管理 → 返回 → 主会话 SPAWN 阶段四
阶段四：警报管理 → 完成
```

每个阶段都是全新的上下文，彻底杜绝"越做越糊涂"的问题。

你可以连续聊一个月，智能体每次分析都是巅峰状态——不会累，不会糊涂。

### 📋 交易周期系统

有人说，最好的交易心态就是永远忘记上一单的输赢。

**我们用"周期系统"实现了这一点。**

每个周期独立管理交易建议：
```
新周期开启 → 给出建议 → 监控止盈止损 → 全部关闭 → 自动归档
```

周期内，智能体会记住：
- 现在有没有持仓？
- 止盈止损设在哪儿？
- 之前为什么给这个建议？

周期结束，自动归档。下一周期从零开始，不受上一轮的影响——**不会因为上一单赚了就激进，也不会因为上一单亏了就畏手畏脚。**

如果你想复盘，归档记录都在那儿，随时可查。但新周期开始时，智能体永远保持清醒和冷静。

## ❓ 常见问题 Q&A

### Q: 什么时候会触发下单？

**A: 只有在日报或即时分析给出交易建议，且建议状态为 `open` 且尚未开仓时，才会自动下单。**

换句话说，不是每次分析都下单。智能体先分析市场，觉得有机会才给建议；给了建议才会去检查是否需要开仓。如果建议是"观望"，那什么都不会发生。

如果你不想启用实盘交易，不配置 OKX CLI 就行了——报告照样写，建议照样给，只是不会真正下单。

### Q: 数据连不上是什么情况？

**A: 国内网络访问 OKX API 需要代理。**

如果你看到类似 `ETIMEDOUT`、`ECONNREFUSED`、或者一直超时的错误，大概率是代理没配置好：
- 检查代理服务是否启动（默认 `127.0.0.1:7890`）
- 检查 `ecosystem.config.js` 里的环境变量是否正确
- 用 `curl --proxy http://127.0.0.1:7890 https://www.okx.com` 测试一下能不能访问

警报器用的是 CryptoCompare，国内可以直接访问。如果警报器也不工作，那就不是代理问题——检查 PM2 日志看具体报错。

### Q: 警报触发了，但没收到即时分析？

**A: 检查智能体配置和 cron 任务。**

警报触发后会调用 `sessions_spawn` 工具唤醒智能体做即时分析。如果没收到：
- 确认智能体 ID 是否正确配置在 cron 任务里
- 确认智能体是否在线运行
- 查看 PM2 日志，看是否有 spawn 相关的错误

### Q: 我想改一下警报规则，怎么弄？

**A: 直接修改或新建 `skills/btc-alert/rules/*.js` 文件。**

警报器支持热加载——你写好规则文件放进 `rules/` 目录，最多 60 秒后就会自动生效。不需要重启 PM2。

如果要删除某个警报，把规则文件移到 `rules-archive/` 目录就行，警报器会自动卸载它。

详细的规则编写方法看 `tasks/set-alert.md`。

### Q: 我不想用飞书，能用其他渠道吗？

**A: 目前飞书是主要支持渠道，但理论上可以扩展。**

报告发送用的是 `message` 工具，支持多种 channel。如果你有 Telegram、Discord 等其他通道配置，可以修改 cron 任务里的 `delivery` 参数，指定目标渠道。

不过这需要你自己配置对应的通道——我们默认只写了飞书的示例。

### Q: 日报太长了，能不能精简一点？

**A: 报告内容是精心设计的，不建议精简。**

日报包含：
- 市场全貌（价格、波动、交易量）
- 技术指标分析（EMA、RSI、斐波那契）
- 交易侧数据解读（资金费率、多空比、OI）
- 期权市场情况（Put/Call 比、Max Pain）
- 交易建议和理由

每一部分都有其分析价值。如果你只想看结论，可以跳到"交易建议"章节——但完整报告能帮你理解"为什么给这个建议"。

### Q: 除了 PM2 和定时任务，我能手动触发日报或即时分析吗？

**A: 当然可以，直接跟智能体说就行。**

不过有个小技巧——**最好新开一个上下文窗口来触发**。

你知道为什么吗？日报和即时分析都要获取数据、写报告、可能还要管理交易建议，整个流程会产生大量日志和数据。如果直接在主会话里执行，这些内容会堆积在你的对话历史里，导致：
- 上下文越来越长，token 烧得飞快
- 后续对话被分析内容"污染"，智能体注意力分散

**正确做法是用 `sessions_spawn` 开一个新会话：**

```
你：七月，spawn 一个新会话做一次日报分析
```

智能体收到后会调用 `sessions_spawn` 工具，创建一个独立会话去执行日报任务。那个会话自己跑完就结束，主会话保持干净——你只会收到一条"日报已完成"的通知，而不是整个分析过程。

**具体触发方式：**

| 任务 | 触发指令 | 说明 |
|------|---------|------|
| 日报分析 | `[SPAWN_DAILY_REPORT]` 前缀 | 或说"spawn 新会话做日报" |
| 即时分析 | 警报自动触发 | 或说"spawn 新会话做即时分析，警报数据是..." |

手动触发即时分析时，你需要自己提供"警报数据"——因为本来是警报器收集数据传过来的。如果你只是想看看当前市场，直接触发日报更合适。

啊，当然，你也可以不用这个前缀，只是使用/new创建一个新窗口，然后问他也行。

## 🚀 5 分钟快速部署

说实话，部署这个技能不难，但要跑起来需要配置几个东西。跟着步骤走就行：

### ① 安装技能

```bash
# ClawHub 安装（推荐）
clawhub install july-btc-analysis

# 或者从 Git 克隆
git clone https://github.com/Dandan044/july-btc-analysis-skill.git
```

### ② 启动警报器引擎

警报器是独立进程，用 PM2 运行：

```bash
cd <技能目录>
pm2 start ecosystem.config.js
pm2 save
```

用 `pm2 logs btc-alert` 可以看到心跳日志——确认它在正常运行。

### ③ 配置定时任务

你希望智能体每天几点给你发日报？通常是早上 9:00 和晚上 21:00。

用 OpenClaw 的 cron 工具添加两个任务：

```json
{
  "name": "btc-morning-report",
  "agentId": "<你的智能体ID>",
  "schedule": { "kind": "cron", "expr": "0 9 * * *", "tz": "Asia/Shanghai" },
  "sessionTarget": "isolated",
  "wakeMode": "now",
  "payload": {
    "kind": "agentTurn",
    "message": "[SPAWN_DAILY_REPORT]执行比特币技术分析日报",
    "thinking": "high",
    "timeoutSeconds": 0
  }
}
```

晚上 21:00 的配置同理，把 `expr` 改成 `0 21 * * *`。

### ④ 配置飞书接收（可选）

日报写完了要发到哪儿？目前支持飞书私聊。

在 `.openclaw/credentials.json` 里配置你的 Open ID：

```json
{ "feishu": { "targetOpenId": "ou_xxx" } }
```

不知道怎么获取 Open ID？飞书开放平台有文档，或者问你的飞书管理员。

### ⑤ 配置 OKX 实盘（可选）

如果你想启用实盘交易功能：

```bash
# 安装 OKX CLI
npm install -g @okx_ai/okx-trade-cli

# 配置 API 密钥
# 编辑 ~/.okx/config.toml，填入你的 API Key、Secret、Passphrase
```

⚠️ **安全提醒**：API Key 只给交易权限，不要给提币权限。

## 🛠️ 警报类型

不只是"价格到了某个位置就触发"，我们还支持更丰富的监控维度：

| 警报类型 | 监控内容 | 为什么有用 |
|---------|---------|-----------|
| **价格警报** | 支撑/阻力位突破 | 关键价位被触碰，意味着行情可能转向 |
| **交易量异动** | 交易量 > 均值×N | 大资金进场/离场，往往是变盘信号 |
| **波动率收窄** | 振幅持续缩小 | 市场在憋大招，变盘前夕要警惕 |
| **情绪极端** | FGI < 10 或 > 80 | 极度恐惧=可能反弹，极度贪婪=可能回调 |
| **定时器警报** | 计划时间触发 | "4小时后检查入场条件"，延迟入场场景 |

**设计原则**：每个方向最多 1 个警报，避免同一价位收到一堆重复通知。
